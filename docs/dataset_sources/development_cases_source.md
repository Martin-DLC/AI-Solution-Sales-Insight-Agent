# Development Cases Source

## DEV-01

### Case ID

DEV-01

### 案例名称

星瀚电商AI客服升级项目

### 开发集/测试集

Development

### 难度

Low

### 行业

综合电商 / 零售

### 企业规模

约600名员工，客服团队约120人

### 当前销售阶段

需求调研阶段，准备进入解决方案交流

### 场景标签

- high_intent
- missing_decision_maker
- weak_data_readiness
- security_risk
- high_delivery_risk

### 客户背景

星瀚电商是一家经营家居、生活用品和小家电的综合电商企业，主要通过天猫、京东、抖音商城和自有小程序开展业务。

公司约有600名员工，其中客服中心约120人，分为售前咨询、订单服务、物流咨询、退换货和投诉处理五个小组。

目前使用第三方在线客服系统和自建订单系统，但客服知识主要分散在飞书文档、Excel、培训材料和历史聊天记录中。不同业务线的客服话术和处理标准不完全一致。

公司希望在今年“双十一”前完成客服AI能力升级，当前距离目标时间约5个月。

### 会议纪要

参与人员：

- 李敏：客户服务中心负责人
- 周凯：电商运营负责人
- 王工：IT架构师
- 陈浩：我方AI解决方案销售
- 刘顾问：我方售前顾问

李敏首先介绍，目前客服中心共有约120人，在大促期间还需要临时增加40到60名外包客服。客服团队日常收到最多的问题是物流进度、退换货规则、优惠活动和商品使用问题。她估计，重复性问题占全部咨询量的一半以上。

李敏表示，公司今年希望重点降低客服成本，同时改善大促期间响应速度。过去两年每次大促都会出现排队时间过长、客服回答不一致以及新人错误回复等问题。

周凯补充，运营团队经常调整优惠活动和商品规则，但客服知识更新速度跟不上。运营部门更新活动后，有时需要两三天才能同步到所有客服人员。因为信息不一致，出现过客户投诉和平台处罚。

陈浩询问公司是否已经确定AI客服项目预算。李敏表示，客服部门已经向管理层提交了预算申请，目前预留范围大约在80万到120万元之间，但最终金额还需要财务负责人和总经理审批。

王工表示，公司现在使用的在线客服系统支持API，但具体开放范围需要和供应商确认。订单系统是公司自建的，可以提供订单状态、物流状态和售后状态查询接口。

王工同时强调，客户聊天记录和订单数据涉及个人信息，不能直接上传到未经安全评估的公共服务中。公司倾向于使用专属实例、私有化部署，或者至少保证数据不被模型训练。

刘顾问询问现有知识库情况。李敏表示，目前没有统一知识库。主要内容分散在约200份飞书文档、几十个Excel文件和大量历史聊天记录中。部分规则已经过期，也存在多个版本。

周凯希望系统不仅能够回答FAQ，还能根据用户咨询推荐适合的产品，并在客户准备离开时发放优惠券。

王工对此比较谨慎。他认为第一阶段应该先解决知识问答和订单查询，不建议一开始就接入自动营销和优惠券发放，因为这会涉及营销规则、价格权限和风控问题。

李敏表示，如果第一阶段效果好，可以先在售前咨询和物流查询两个业务组试点，试点人数约20人。她希望系统能够在三个月左右完成试点，并在“双十一”前决定是否扩大使用范围。

陈浩询问客户如何判断试点是否成功。李敏回答，希望至少能够减少人工处理的重复问题，同时缩短平均响应时间，但目前还没有明确的量化指标。

会议最后，李敏希望我方在一周内提供一份初步解决方案，包括项目范围、技术路线、实施周期、报价范围和试点建议。王工要求下一次会议必须重点讨论部署方式、数据脱敏、系统接口和权限控制。

### 销售备注

李敏在会后单独告诉我，她个人非常支持这个项目，并认为管理层大概率会批准。她希望我方方案尽量写得有吸引力，最好能够承诺替代30%以上的客服人员。

我判断客户预算明确、时间明确，应该属于高质量商机。建议尽快提交完整报价，并推动客户直接签署POC合同。

客户之前也接触过两家AI客服厂商，但没有透露具体厂商和报价。

### 已知限制

- 最终预算尚未获得财务和总经理审批
- 在线客服系统API能力尚未确认
- 知识文档存在过期和多版本问题
- 客户数据涉及个人信息
- 试点验收指标尚未量化
- 竞品情况不清楚
- 采购和法务流程尚未了解
- 第一阶段是否允许接入大模型外部API尚未确定

### 可用方案库

- AI客服知识问答方案
- RAG企业知识库方案
- 订单与物流查询Tool方案
- 客服辅助回复方案
- 客服质检分析方案
- 商品推荐Agent方案
- 营销优惠券自动发放方案
- 私有化大模型部署方案
- 专属云实例部署方案

## DEV-04

### Case ID

DEV-04

### 案例名称

悦庭零售集团AI导购与会员运营咨询项目

### 开发集/测试集

Development

### 难度

Medium

### 行业

连锁零售 / 家居生活

### 企业规模

约1800名员工，拥有85家线下门店及自营电商渠道

### 当前销售阶段

初次需求交流 / 市场调研阶段

### 场景标签

- low_intent
- missing_budget
- missing_decision_maker
- ambiguous_requirement
- weak_data_readiness
- unrealistic_scope
- cross_sell_opportunity

### 客户背景

悦庭零售集团主要经营家居用品、生活百货和小型家电，在华东地区拥有85家线下门店，同时运营天猫旗舰店、自营小程序和企业微信私域社群。

集团近年来希望推进会员数字化运营，但当前线下门店、电商平台和私域渠道分别使用不同系统。会员信息、消费记录和营销活动数据尚未完全打通。

集团董事长近期在行业会议上关注到“AI导购”和“AI员工”概念，要求数字化团队了解相关解决方案，但目前尚未形成正式项目。

### 会议纪要

参与人员：

- 张雯：集团数字化创新经理
- 孙倩：会员运营负责人
- 何工：IT应用负责人
- 高峰：区域门店运营经理
- 陈浩：我方AI解决方案销售
- 刘顾问：我方售前顾问

张雯介绍，董事长近期多次提到AI，希望集团能够尽快探索一个能够对外展示的AI项目。数字化团队因此开始接触不同厂商，目前已经沟通过约五家AI服务商。

她表示，希望未来消费者进入小程序后，可以直接和AI导购对话，获取商品推荐、搭配建议、门店库存、优惠活动和售后政策等信息。她还希望AI能够根据会员历史消费记录自动生成营销内容，并通过企业微信完成客户触达。

孙倩补充，目前会员运营团队每个月需要手动设计大量营销活动。不同渠道之间的数据并不完整，门店会员手机号、电商会员账号和企业微信客户身份经常无法对应。

她认为AI如果能够帮助分析会员偏好、自动推荐商品并生成营销文案，可能会提升复购率。但目前团队没有明确统计客户推荐转化率，也没有统一的会员标签体系。

高峰表示，门店导购对AI项目比较感兴趣，因为新人熟悉商品通常需要两到三个月。但不同门店经营品类、促销政策和库存情况存在差异。如果系统推荐错误商品或错误优惠，可能引起客户投诉。

何工说明，目前商品主数据存放在ERP中，库存数据每两小时同步一次。会员数据分别存在CRM、电商平台和企业微信服务商系统中。集团尚未完成统一会员ID建设。

他还表示，现阶段不确定现有系统能否稳定开放接口，也没有对大模型使用客户消费数据进行安全评估。

陈浩询问项目预算和上线时间。张雯表示，目前主要是响应董事长要求开展调研，还没有正式预算。她计划先收集几家厂商的方案，再向集团管理层汇报。

孙倩希望最好能在两个月内做出一个能演示的版本，但是否真正上线，需要等管理层决定。她也没有明确由哪个部门承担项目费用。

刘顾问询问第一阶段最重要的业务目标。张雯认为可以同时做AI导购、会员分析、营销文案、智能客服和门店员工培训，这样展示效果更好。

何工认为范围太大，建议先确定一个场景，并先解决商品数据、会员身份和接口问题。

会议最后，张雯要求我方在下周提供一份完整解决方案，包括整体蓝图、产品Demo、实施路线和费用预估，用于管理层内部讨论。

### 销售备注

张雯在整个会议中表现得非常积极，而且董事长亲自关注AI，我认为这是集团级战略项目。

虽然客户暂时没有预算，但如果我们的方案足够有冲击力，项目金额可能超过200万元。

建议我们直接准备一个覆盖AI导购、会员运营、智能客服和员工培训的整体方案，并尽快推动免费POC，以便抢在其他五家竞争对手前面。

### 已知限制

- 客户没有正式立项
- 没有明确预算
- 没有明确费用承担部门
- 最终决策人没有参与会议
- 董事长的具体目标和优先级不明确
- 统一会员ID尚未建立
- 多渠道会员数据没有打通
- 接口能力尚未确认
- 数据安全评估尚未开展
- 第一阶段业务目标没有统一
- 已有约五家竞争厂商参与
- 客户提出的两个月时间仅针对演示，不等于上线承诺

### 可用方案库

- AI商品导购方案
- 商品知识库RAG方案
- 会员运营分析方案
- 营销内容生成方案
- AI客服知识问答方案
- 门店员工知识助手方案
- 企业数据治理咨询方案
- 客户身份统一与数据集成方案
- 轻量级产品Demo方案

## DEV-05

### Case ID

DEV-05

### 案例名称

华岳工业设备售后服务Agent项目

### 开发集/测试集

Development

### 难度

High

### 行业

工业设备制造 / 售后服务

### 企业规模

约2500名员工，售后服务人员约180人

### 当前销售阶段

初步需求调研阶段

### 场景标签

- weak_process_readiness
- weak_data_readiness
- solution_mismatch
- high_delivery_risk
- stakeholder_conflict
- unrealistic_scope
- not_suitable_for_ai

### 客户背景

华岳工业设备主要生产包装机械、输送设备和自动化产线，客户分布在全国多个区域。

公司拥有约180名售后服务人员，包括400客服、区域服务工程师、总部技术专家和备件管理人员。

目前客户通过电话、微信、区域销售人员和400热线提交问题。不同渠道的信息最终由人工转发给对应区域工程师。

公司没有统一的售后工单平台，服务过程主要记录在Excel、微信群、个人笔记和ERP维修备注中。

### 会议纪要

参与人员：

- 郑强：售后服务总监
- 林雪：400客服主管
- 赵经理：华东区域服务经理
- 彭工：总部技术专家
- 吴工：信息化负责人
- 陈浩：我方AI解决方案销售
- 刘顾问：我方售前顾问

郑强介绍，目前售后团队工作量很大，希望通过Agent自动接收客户问题、判断故障类型、分配工程师、查询备件、生成维修方案并通知客户。

他希望Agent能够替代大部分400客服，并减少区域经理人工协调工作。

林雪表示，目前客户报修时提供的信息经常不完整。有些客户只说“机器不能用了”，客服需要反复询问设备型号、故障代码、现场照片和停机情况。

不同客服人员的提问方式和记录内容并不一致。

赵经理补充，不同区域的派工方式不同。华东区域按客户等级和工程师技能分配，华南区域更看重地理距离，西部区域则经常由销售人员临时协调。

公司没有统一的工程师技能标签，也没有实时更新工程师的位置和工作状态。

彭工表示，设备故障处理高度依赖经验。同一种报警代码可能由多种原因造成，必须结合设备版本、运行环境、维修历史和现场照片判断。

现有维修案例主要保存在个人电脑、微信群和纸质记录中，没有统一模板。

郑强认为这些问题正说明需要Agent。他希望系统可以自动学习历史聊天和维修记录，并直接给客户维修建议。

吴工对此表示担忧。公司目前连统一工单系统都没有，ERP中的设备编号也经常和销售系统中的客户名称对应不上。

部分老设备没有完整电子档案，维修记录的数据质量很差。

刘顾问询问售后流程是否有统一SOP。林雪表示400客服有一份基础登记表，但各区域实际处理方式差异很大。

赵经理认为总部统一SOP不一定适合所有区域，因为客户等级和服务合同不同。

郑强要求第一阶段尽量做到自动受理、自动诊断、自动派单和自动生成解决方案。他希望三个月内上线，并减少至少40%的客服和协调人员。

吴工表示，目前公司没有批准新的售后系统预算，也没有确定哪个系统作为未来工单主系统。

会议最后，郑强要求我方直接提供一套多Agent售后服务平台方案，并展示自动诊断和自动派工能力。

### 销售备注

客户痛点非常明显，而且售后总监明确希望减少40%的人员，我认为这是一个很大的降本项目。

虽然流程和数据比较混乱，但这正是大模型擅长处理的非结构化场景。

建议直接设计多个Agent：

- 客户接待Agent
- 故障诊断Agent
- 派工Agent
- 备件Agent
- 客户通知Agent

可以先用大模型读取微信群和Excel数据，通过多Agent协作直接替代现有流程。

### 已知限制

- 没有统一售后工单系统
- 报修入口分散
- 售后流程没有统一SOP
- 不同区域派工规则不同
- 没有统一工程师技能标签
- 没有实时工程师状态数据
- 设备、客户和维修数据无法稳定关联
- 历史维修知识高度分散
- 数据质量差
- 故障诊断依赖现场环境和专家经验
- 当前没有正式预算
- 没有确定未来系统Owner
- 三个月上线范围不现实
- 人员替代目标没有数据依据

### 可用方案库

- 售后知识库RAG方案
- 客户报修信息采集助手
- 工单摘要生成方案
- 故障案例检索助手
- 工程师辅助诊断方案
- 服务工单系统集成方案
- 自动派工Agent方案
- 备件查询Tool方案
- 客户通知Agent方案
- 售后流程梳理与数据治理咨询方案
